# 记忆增长的成本/延迟 vs 收益 (ROI) 分析

> 自进化 Agent 靠经验池 (episodic memory) 变大提升解决率,但每多一条经验,
> 每个 query 注入的检索上下文也变多 -> token / 成本 / 检索延迟同步上升。
> 本报告把收益曲线和成本/延迟曲线对齐,量化 ROI 并找出边际收益递减拐点。

## 4. 三个拐点 (trade-off 的真实形状)

这条曲线不是简单的收益平台,而呈现三段,每段对应一个工程动作:

**(A) 冷启动死区 (round 0→2, pool 0→3)**:经验池在长大、注入 token 从 411.5 涨到 467.7(+56 token),但存的经验还匹配不上 eval 的 paraphrase,**解决率纹丝不动 (34.2%)**。这是纯加价、零收益区——工程上要尽快跳过 (预热 / 主动检索而非被动堆案例)。

**(B) 成本效率甜区 + 拐点 (knee = round 4, pool=17)**:cost-efficiency knee: marginal cost is cheapest here (0.0 tokens per +1% resolution); beyond this, latency keeps rising while token/hit gains are capped by top-k, so each extra point of resolution costs more.

**(C) token 饱和点 (round 4, pool=17)**:注入 token / 检索命中受 **top-k 上限**钳制,在 ~519 token / 6.92 hits 处饱和;此后解决率还能涨 (靠命中质量更优,而非更多上下文),但**延迟仍随经验池线性上升** (0.2129→0.2373 ms,+11%)——这之后的成本主要来自 BM25 检索规模而非 token。

> 关键洞察:**token 早早饱和 (top-k 钳制),延迟却一直涨**——所以在规模化时,**延迟/检索规模**才是比 token 更敏感的成本代理,记忆淘汰要盯的是池子大小,不只是上下文长度。

## 5. 工程结论 & 面试话术

- **预算钉在拐点附近**:用 retrieval **top-k 截断** + episodic 记忆 **TTL / LRU 淘汰**,
  把经验池有效规模控制在拐点 (pool≈17) 量级,而非无限增长。
- **拐点后改质不改量**:与其堆经验,不如做去重 / 反思蒸馏 (playbook) 压缩,提升单位 token 的信息密度。
- 监控指标:把 `avg_inj_tokens`、`avg_hits`、`avg_latency_ms`(本仓 `obs.aggregate` 已有)接看板,
  对 token 设 SLO,越拐点即告警。

> 面试话术: 我不会让记忆无限膨胀——我先量化了 ROI: 解决率从 34% 涨到 71% 的同时, 
> 注入 token 涨了 27%、延迟涨了 63%;而且 token 在 pool≈17 就被 top-k 钳到饱和,延迟却一直随池子线性涨。
> 所以我会用 **TTL + top-k 截断 + 反思蒸馏**把成本控制在效率拐点附近 (pool≈17),并优先按检索规模而非上下文长度做记忆淘汰——而不是为最后几个点去无脑堆经验、付延迟的账。
